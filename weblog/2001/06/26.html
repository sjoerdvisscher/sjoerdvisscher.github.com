Scripting vs Programming

(Te be clear: with programming I mean languages like C and Java.)

Programming is about creating programs that run efficiently.
Scripting is about efficiently creating programs that run.
As a result of that, programming is about manipulating data in computer formats (ie. memory blocks), whereas scripting is about manipulating data in "real world" form, like numbers, text, lists and dictionaries.

So ideally scripting languages shouldn't have different types that have different in-memory format, but the same meaning to a scripter. And indeed most scripting languages don't have byte and word and dword datatypes, but just one 'number' type. However, most of them do have both records and dictionaries, while to a scripter they both are objectshttps://w3future.comhttps://w3future.comhttps://w3future.comhttps://w3future.com with named fields.

One might argue that a scripter can always use the more general datatype, and switch to a faster and more specific datatype when needed. However a lot of built-in functions only allow one datatype, leaving the scripter no choice.

What I find even more disappointing is that I've yet to find a scripting language that has built-in support for the most popular real world data format today: the attributed tree (ie. a model like XML). And with this I don't mean built-in XML support, with access to the data through some (usually clumsy) API. I mean support for objects that allow access to attributes like you would with records or dictionaries, like myObj.name="test", and access to it's children like you would with lists, like foreach child in myObj.

In fact, the attributed tree object can be the only built-in type a scripting language would need, besides the scalar types string, date, boolean, float and number. Javascript comes close, it implements it's arrays as dictionaries, ie. the 0th element with key "0", the 1st element with key "1", etc. This way it's possible to add properties (attributes) to the arrays (lists). But it's hard and slow to manipulate such lists, and you can't use for..in, which iterates through all keys.

If somebody knows a scripting language that does have proper support for attributed trees, or if there are some good papers on this subject, please let me know.
